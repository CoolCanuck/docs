    Developer's Code of Conduct

    The purpose of this document is to inform developers of their rights and responsibilities and lay out the rules for fair play.

    See Also: Users Code of Conduct.

    Golden Ruleslink

    1. Respect your fellow developer and communicate through any issues you have civilly.
    2. Make room for other developers to get a chance to work on games they like, in their own way.
    3. Anyone can become a developer if they put in the effort.
    4. All achievement sets are open for improvement.

    How to Become an Achievement Developerlink

    1. Start by learning about creating achievements. Optional but recommended: join our Discord server and ask a mod to give you the jr-dev role. Use #help-me and #devs channels any time you have questions.

    2. Follow the guidelines below to find a game to work on achievements for it.

    3. Post your plan for an achievement set on the official game's forum topic, so the community can leave suggestions and provide feedback.

    4. Create some achievements and promote them to Unofficial.

    5. Contact a mod on Discord (or send a message to RAdmin if you don't use Discord) and ask them to find someone to review your work. Please, be open to receive feedback and suggestions.

    6. Once the community reviewed your code and achievement concepts, the admins will use their judgement to approve or deny developer status and tell you what you'll need to do if you're not ready.

    Basic Achievement Design Guidelineslink

    Unwelcome Conceptslink

    You can (and should) be as creative as you can, but there are some concepts that are NOT welcome, such as:

    • Achievements obtainable with zero effort for no meaningful purpose (example: start the game as character X, collect one coin). It's acceptable if there's something fun/historical/interesting you want to address, but make it worth 1 or 2 points.

    • Achievements that require glitches (acceptable in Bonus sets).

    • Achievements that require perfection all game (maybe acceptable in Bonus Sets).

    • Achievements for dying (simply), or playing bad without any purpose.

    See also: Set Development Roadmap and Achievement Design.

    Every Achievement Set MUST Havelink

    • Protection for situations where the players can get achievements without effort, such as:
    • Ingame cheat codes;
    • Battery saves;
    • Passwords

    See also: Achievement Templates and Real Examples for some well known protection techniques.

    Every Achievement Set SHOULD Havelink

    • At least basic Rich Presence.

    • Game Badges for each achievement.

    • All game page information filled out, and game images uploaded.

    Note: Rich Presence, Game page information and images, and leaderboards cannot be added until you are a full developer.

    Working on Empty Setslink

    If you find a game you are interested in creating achievements for, you first need to check if it already has an achievement set.

    If the game has a set and you want to improve it, you should go through the revision process.

    If the game is not even present on the website's database, ask for it to be added on Discord (#help-me) or on our forums and some dev will add it for you. Note: prefer to use no-intro ROMs.

    If you want to work on a ROM hack, check this doc first: Achievements for ROM hacks.

    If the game has no achievements, you must check if there's someone working on it before doing anything. The reason is to avoid conflicts between multiple devs working on the same game at the same time without knowing about it.

    Recommended ways to check if there's someone working on a game:

    • The comments on the game's page.
    • Posts on the game's official forum topic (you can find the link on the game's page).
    • Someone already left some Code Notes.

    If you have found a dev who is already working on a set for the game you are interested in you must post your desire to work on the set publicly, in the forum or in the Discord developer channel, and then work it out with the existing developer.

    If the dev doesn't answer you in a couple of days, you can go ahead.

    So, the game is free for you to worklink

    We have a roadmap you can use as a guideline to create a good achievement set: Set Development Roadmap. We also have a guide on how to design good achievements, not the technical side but the conceptual: Achievement Design.

    Now that you've found a game to work on, before doing any work you must do the following:

    1. Leave a comment on the game's page saying that you're going to work on a set for it.

    2. Post your plan on the official game's forum topic, so the community can leave suggestions and provide feedback.

    Important Note: If you don't agree with the feedback received, you can go ahead with your concept. But keep the following in mind:

    • be sure to addhere the Basic Achievement Design Guidelines, otherwise your set won't be approved.
    • once you published your work, you are giving it to the community to be reviewed and reworked over time.

    Work-In-Progresslink

    Work-In-Progress means you are actively working on a set and are notifying the community achievements are coming. A Work-In-Progress set is locked down for you to work on it.

    It is preferable and recommended to work on one set at a time. But if you have more, be open for a teamwork or even delegate the set creation if another dev contact you.

    If you're no longer working on a set, please remove your Work-In-Progress status.

    If a Work-In-Progress game appears abandoned (has been not worked on for ~2 months or more) and you'd like to work on it attempt contact the developer and work it out with them before you begin. If they are unresponsive within a few days, feel free to proceed. Send a PM to RAdmin if you have any issues.

    Revisions - Working on Sets with Existing Achievementslink

    Preparing your planlink

    Working on a set with existing achievements is called a revision. Any developer wanting to revise a set need to:

    1. prepare a plan for each
    2. Addition
    3. Demotion
    4. Achievement rework

    5. Change to leaderboards

    6. Present the plan

    7. In the game page forum thread in full
    8. In #revision-voting post a link to it
    9. Notify the original developer(s), (no response is needed)

    Getting Community Approvallink

    Anyone in the community can express their concerns or objections. Allow time for discussion and for the developer to adjust their plan if they decide to. If any concerns or objections were expressed, regardless of the developer adjusting the plan, a vote will take place.

    Once the discussion has settled a vote should be posted by a developer in the #revision-voting channel.

    The vote is approved/rejected after 72 hours. It is pass/fail by simple majority.

    Guidelineslink

    • Be very cautious with demotions, otherwise players will upset some players.

    • If you are adding a similar concept to one found in the set, reuse the old concept and update it rather than making a new one (to avoid demotions).

    • Do not code new achievements that are mirrors of old ones.

    No need for approval tolink

    • Update badges (But post old badges side-by-side to new in forum.)

    • Fix bugs

    • Update rich presence (But explain changes or post backup of previous in forum.)

    • Add battery save, cheat code, or password protections to other's work

    • Make changes to game page information and images (But make backup of previous images in forum.)

    Code Noteslink

    You're free to add any code notes you discover to any set without declaring intentions to work on the game. Just be careful to not delete previous notes added by someone else.

    Handling Ticketslink

    After you've published a set or achievements be prepared for bug reports.

    You're expected to keep your work bug free by appropriately resolving tickets. There are many reason players create tickets: Discovering actual bugs, desyncs (when the connection to the server is disrupted), the player is using the wrong ROM version, or the player doesn't understand the achievement.

    Respond to all tickets as soon as possible, ideally within a week. This is important as the problem is fresh in the player's mind and you can use them to help discover and resolve the problem.

    While resolving all tickets, leave a brief summary of how you resolved the player's report. When you're unable to resolve the problem or have indication of false report leave a message showing what conclusions you've made, and then close the ticket. If the problem resurfaces you'll be able to proceed from where you left off.

    Achievement/Set Ownershiplink

    When you publish work you are giving it over to the community to be reviewed and reworked over time.

    Feedback about the Developer's Code of Conductlink

    Although the Developer's Code of Conduct is the result of an intense debate, it's not an immutable document. If you have suggestions for improvements, contact us on our Discord server or send a message to RAdmin.
